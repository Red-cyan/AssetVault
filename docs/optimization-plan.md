# AssetVault 优化计划

## 一、当前结论

AssetVault 目前更接近“传统素材管理后台 + AI 功能按钮”，还不能称为 Agent 项目。

主要问题：

- 智能分析只把文件名、路径、扩展名发给文本模型，没有读取图片、视频帧或 3D 预览内容。
- 自然语言搜索是固定同义词加 SQL `LIKE`，不是 Embedding 语义检索。
- 没有 Agent 的规划、工具调用、执行确认、状态记录和结果反馈闭环。
- 素材库页面功能堆积，添加目录、扫描、搜索、批处理、AI 分析集中在一个页面。
- 扫描时会重新计算指纹和缩略图，大目录下性能较差。
- 扫描任务依赖 FastAPI 进程内后台任务，服务重启后任务会丢失。
- AI 调用失败后静默降级为规则结果，用户不知道模型是否真正调用成功。
- 文档中的 PostgreSQL、Redis、Celery、Embedding 与实际实现存在差距，简历表述容易被追问。

## 二、优化目标

将项目定位调整为：

> 面向本地数字资产的智能管理平台，通过持久化任务系统完成素材索引，利用多模态分析和混合检索理解素材，并由工具调用型 Agent 执行搜索、整理和项目组装任务。

最终形成以下闭环：

```text
目录同步
  -> 元数据与预览提取
  -> 多模态 AI 分析
  -> 关键词 + 向量混合检索
  -> Agent 理解用户目标
  -> 调用搜索、标签、项目等工具
  -> 用户确认写操作
  -> 保存执行轨迹与结果
```

## 三、P0：先解决基础可用性

预计用时：3～5 天。

### 1. 简化本地启动与登录体验（已完成）

- 默认采用免登录本地工作区，首次启动自动创建或复用唯一用户。
- 保留可选密码认证模式，用于远程部署和用户数据隔离演示。
- 前端统一处理运行模式、会话校验和过期 Token，不在业务页面重复跳转。
- 本地模式隐藏登录、退出、邮箱和密码功能，并限制服务默认只监听本机。

### 2. 重构素材库交互

- 素材库只保留浏览、筛选、搜索和详情。
- 将目录管理移动到独立“数据源”页面。
- 将任务进度统一放入任务中心。
- AI 分析增加 loading、取消、失败和来源状态。
- 删除、批量打标签等操作增加确认和结果反馈。
- 保存筛选条件、视图模式和每页数量。
- 优化空状态、错误提示和首次使用流程。

### 3. 修复扫描性能

- 通过 `size + mtime` 判断文件是否变化。
- 文件未变化时跳过哈希、缩略图和元数据重建。
- 遍历文件时流式处理，不一次性保存全部路径。
- 单文件失败不终止整个任务。
- 保存失败文件、失败原因和重试次数。
- 防止同一目录被并发扫描。
- 增加扫描取消和重新执行能力。

### 4. 明确本地目录部署方式

浏览器不能直接让 Docker 后端访问任意 Windows 本地路径，因此需要明确选择：

- 秋招演示阶段优先采用“前后端本机运行”。
- Docker 模式通过挂载目录后扫描容器内路径。
- 文档明确宿主机路径和容器路径映射关系。
- 后续有余力再封装成 Tauri/Electron 桌面应用。

验收标准：1 万个素材可以重复扫描，未变化文件不重复生成指纹和缩略图，单文件损坏不会导致整个任务卡死。

## 四、P1：把 AI 功能做真实

预计用时：5～7 天。

### 1. 多模态素材分析

不同文件采用不同分析输入：

- 图片：原图或压缩图输入视觉模型。
- 视频：抽取关键帧后输入视觉模型。
- 3D 模型：生成预览截图后分析。
- 动作和 UE 文件：结合文件名、目录、格式和可提取元数据。
- 不支持预览的文件明确标记为“仅元数据分析”。

AI 输出使用结构化 Schema：

```json
{
  "summary": "演唱会主舞台模型",
  "category": "scene",
  "tags": ["舞台", "演唱会", "LED"],
  "style": ["现代", "科技"],
  "colors": ["黑色", "蓝色"],
  "confidence": 0.87
}
```

### 2. 避免静默降级

- 记录 `model`、耗时、Token、错误和分析来源。
- 前端明确展示“视觉模型分析”或“本地规则分析”。
- AI 调用失败时允许重试，不用本地结果伪装成 AI 结果。
- 新生成标签先作为建议，由用户确认后写入正式标签。

### 3. 实现真正的语义检索

- 为素材描述、标签和可靠提取内容生成 Embedding，不对只有路径和扩展名的文件生成向量。
- 使用 PostgreSQL + pgvector，或 MVP 阶段使用本地向量库。
- 实现关键词检索与向量检索的混合排序。
- 支持类型、标签、目录、评分等结构化过滤。
- 增加“相似素材”接口。

验收标准：准备 30～50 条固定查询，比较规则搜索和混合检索的 `Recall@10`，用数据说明优化效果。

### 4. 异构格式解析（已完成基础框架）

- 使用 Extractor 注册表按格式提取结构化元数据，解析失败不影响目录扫描。
- PMX 提取内嵌模型名与注释；VMD 提取骨骼、Morph、相机关键帧和时长，但不猜测动作含义。
- OBJ 提取几何统计、对象、分组、材质和 MTL/贴图依赖，glTF/GLB 提取场景、节点、网格、材质、动画和外部依赖。
- FBX 提取容器、版本及可读对象名；Blend 提取文件头；UProject 提取引擎、模块、插件和明确描述。
- `.casc`、`.uasset` 等暂时无法可靠解释的文件标记为“仅元数据”。
- 资产详情展示解析来源、状态、错误、格式元数据和语义索引资格。
- 提取器带版本标识，解析器升级后可对未修改文件补做解析而不重复计算哈希和缩略图。
- 提取出的结构名与依赖名进入关键词检索，但仅有技术元数据时仍禁止生成语义向量。

## 五、P2：引入真正的 Agent

预计用时：5～7 天。

Agent 不建议做成普通聊天框，应围绕素材工作流提供工具调用。

### 1. Agent 工具

```text
search_assets
find_similar_assets
get_asset_detail
suggest_tags
apply_tags
create_project
add_assets_to_project
find_duplicates
inspect_missing_assets
get_task_status
```

### 2. 执行流程

```text
用户目标
  -> 识别意图和约束
  -> 生成执行计划
  -> 调用只读搜索工具
  -> 汇总候选素材
  -> 请求用户确认写操作
  -> 调用标签/项目工具
  -> 返回结果和执行记录
```

示例任务：

> 找出适合赛博朋克演唱会的舞台、角色和动作素材，分别选择 3 个，创建一个项目并分类。

Agent 应当能够分解任务、分别检索、解释选择理由，在用户确认后创建项目。

### 3. Agent 安全边界

- 搜索、查看等只读操作可以自动执行。
- 打标签、创建项目需要确认。
- 删除和批量修改必须二次确认。
- 每次工具调用记录输入、输出、耗时和错误。
- 限制最大执行步数，避免循环调用。

## 六、P3：补齐后端工程能力

预计用时：4～6 天。

- 使用 Alembic 管理数据库迁移。
- 将扫描、缩略图、Embedding、AI 分析迁移到持久化任务队列。
- 秋招项目可选 Redis + Celery，也可以采用更轻量的数据库任务队列。
- 增加任务重试、超时、取消、幂等和服务重启恢复。
- 为常用查询增加组合索引。
- API Key 加密保存，响应和日志禁止输出密钥。
- 增加结构化日志和请求 ID。
- 处理历史 `pending/running` 僵尸任务。

## 七、测试与评价体系

当前 22 个测试通过，但主要覆盖接口成功路径，还需要增加：

- 文件扫描异常、权限不足和文件扫描中被删除。
- 重复扫描幂等性和增量扫描性能。
- AI 超时、无效 JSON、限流和重试。
- Agent 工具权限与写操作确认。
- 多用户资源隔离。
- 任务重启恢复。
- 真实前端核心流程 E2E 测试。
- 语义检索离线评测集。

## 八、推荐实施顺序

1. 素材库交互和扫描稳定性。
2. 多模态分析。
3. Embedding 混合检索。
4. 工具调用型 Agent。
5. 持久化任务队列和数据库迁移。
6. 测试、指标、演示数据和简历文档。

不建议先花时间做复杂权限、云同步、3D 在线编辑器或微服务拆分，这些不能解决当前“不好用”和“Agent 不真实”的核心问题。

## 九、简历表述建议

完成 P0～P2 后，可以写成：

> 设计并实现面向数字内容创作者的本地智能资产管理平台，基于 FastAPI、Next.js 和 PostgreSQL 构建素材索引与任务系统；通过多模态模型提取素材语义信息，结合 pgvector 与关键词检索实现混合搜索；设计工具调用型 Agent，将自然语言目标分解为素材检索、相似推荐、标签整理和项目组装操作，并通过执行确认和轨迹记录保障写操作安全。

在这些能力真正完成前，不建议在简历中写“基于 Agent/LangGraph”“向量检索”“Celery 分布式任务”等尚未落地的内容。
