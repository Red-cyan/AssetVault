# AssetVault 面试讲解稿

本文档用于秋招面试、项目答辩和简历追问。目标不是把所有代码细节背下来，而是把项目定位、架构取舍、核心难点和可扩展方向讲清楚。

## 1. 项目一句话

AssetVault 是一个面向 UE5、Blender、MMD 创作者的本地 AI 素材管理平台。它不移动用户原始文件，而是扫描本地目录建立数据库索引，提供素材浏览、标签管理、缩略图预览、项目引用、重复检测、失效检查、回收站、统计分析和 AI 辅助检索能力。

可以这样介绍：

> 我做 AssetVault 的原因是数字内容创作者经常会有大量分散在本地硬盘里的模型、贴图、视频、动作文件。传统文件夹管理很难搜索、分类和复用。所以我把它设计成一个“只建索引、不改原文件”的素材管理系统，用数据库维护元数据，用缩略图和标签提升浏览效率，再用 AI/语义搜索降低查找成本。

## 2. 项目定位

这个项目不是简单的网盘，也不是文件上传系统，而是本地数字资产索引器。

核心定位：

- 文件仍然保存在用户原始目录，系统只保存索引和元数据。
- 面向素材复用场景，不面向普通文档管理。
- 搜索、标签、缩略图、项目引用和 AI 分析都围绕“如何更快找到素材”展开。
- 对 UE5、Blender、MMD 常见素材格式做了类型识别，包括图片、视频、PMX、FBX、OBJ、GLB、Blend、VMD、UAsset 等。

和 PitWallAgent 的区别：

- PitWallAgent 更偏 AI 工程和 Agent/RAG。
- AssetVault 更偏企业级全栈产品，包含用户系统、数据库设计、文件处理、异步任务思路、前后端协作、部署和工程文档。

## 3. 技术栈

后端：

- FastAPI：提供 REST API，适合快速构建类型清晰的后端服务。
- SQLAlchemy：维护实体模型、查询和关系。
- SQLite：当前作为本地开发数据库，后续可以迁移到 PostgreSQL。
- Pydantic：请求和响应模型校验。
- Pillow：图片缩略图生成。
- FFmpeg：如果本机安装，则用于视频缩略图生成。
- Pytest + Ruff：后端测试和代码质量检查。

前端：

- Next.js + TypeScript：实现素材库、项目、统计、设置等页面。
- 原生 CSS/组件组织：当前优先保证功能闭环和演示稳定性。
- 前端通过统一 API Client 和后端通信。

部署：

- Dockerfile 分别构建前后端。
- Docker Compose 编排前端、后端、数据库文件卷、缓存卷和备份卷。
- `.env.example` 提供本地部署配置模板。

认证采用双运行模式：本地桌面式使用默认免登录并自动绑定单一工作区；需要远程访问时
切换到密码模式，使用 JWT 和 `user_id` 完成数据库业务数据隔离。保留用户外键让两种模式
共用同一套模型，也兼容已有数据。这里需要明确说明，JWT 隔离不是操作系统文件权限隔离，
远程部署还必须限制后端可读取的挂载目录。

## 4. 当前功能闭环

MVP 已经覆盖了素材管理的完整流程：

1. 进入本地工作区（远程密码模式下先登录）
2. 添加素材目录
3. 扫描本地目录
4. 建立素材索引
5. 浏览网格/列表
6. 按名称、标签、类型、路径等搜索
7. 查看素材详情
8. 编辑标签、评分、收藏、备注
9. 生成图片/视频缩略图
10. 创建项目并引用素材
11. 查看统计面板
12. 执行智能分析和自然语言搜索
13. 检测重复素材
14. 检测失效素材
15. 使用回收站恢复或永久删除索引
16. 备份数据库

面试时可以强调：这不是单点 Demo，而是一个可日常使用的软件闭环。

## 5. 后端架构

后端按典型分层组织：

- `api/v1`：负责 HTTP 路由、参数接收、权限依赖和响应返回。
- `schemas`：负责 Pydantic 请求/响应结构。
- `models`：负责 SQLAlchemy 数据模型。
- `services`：负责业务逻辑，例如扫描、缩略图、搜索、重复检测、备份等。
- `core`：负责配置和安全能力。
- `db`：负责数据库连接和 Base 声明。
- `tests`：覆盖核心 API 行为。

可以这样讲：

> 我没有把业务逻辑堆在 FastAPI 路由里，而是把扫描、缩略图、重复检测、搜索等能力拆到 service 层。这样 API 层只负责协议，service 层可以被测试和复用，也方便以后把耗时任务迁移到 Celery 或后台队列。

## 6. 数据模型设计

核心表控制在可维护范围内：

- `users`：用户账号和认证信息。
- `folders`：用户配置的素材目录。
- `assets`：素材索引和元数据。
- `tags`：标签。
- `asset_tags`：素材和标签多对多关系。
- `projects`：项目。
- `project_assets`：项目引用的素材。
- `settings`：系统和 AI 配置。
- `tasks`：扫描、分析等任务记录。

设计重点：

- 素材表保存路径、大小、类型、时间、评分、收藏、描述等元数据。
- 多标签通过中间表实现，避免把标签拼接成字符串。
- 项目引用和素材本体解耦，一个素材可以被多个项目复用。
- 回收站只软删除数据库索引，不删除磁盘文件，降低误删风险。

## 7. 核心难点与解决方案

### 7.1 不移动原文件

难点：

如果系统直接管理文件移动和删除，容易造成用户资产丢失，也会带来跨盘符、权限和路径更新问题。

方案：

AssetVault 只扫描目录并保存索引。删除默认进入数据库回收站，只修改索引状态，不删除真实文件。永久删除也只是删除数据库索引。

面试表达：

> 这个取舍牺牲了一部分文件整理能力，但换来了更高的数据安全性，也更符合本地素材库的使用习惯。

### 7.2 高速搜索

难点：

Windows 文件搜索对大量素材目录不稳定，而且很难把标签、备注、项目引用等业务元数据纳入搜索。

方案：

系统扫描时把文件元数据写入数据库，搜索时查数据库索引，而不是每次遍历磁盘。当前支持名称、标签、作者、类型、路径等维度，后续可以加 PostgreSQL 全文索引或向量检索。

### 7.3 缩略图生成

难点：

素材文件类型很多，图片、视频、模型的预览方式不同。视频还依赖 FFmpeg 环境。

方案：

当前先覆盖高价值的图片和视频：

- 图片用 Pillow 生成缩略图。
- 视频在检测到 FFmpeg 后截帧生成缩略图。
- 模型缩略图作为第二阶段扩展，可以接 Blender headless 或 three.js 渲染服务。

### 7.4 重复检测

难点：

完整计算大文件 hash 成本高，素材库里可能有大体积视频和模型文件。

方案：

当前使用快速文件指纹，结合文件大小进行重复判断。这样适合 MVP 阶段快速发现明显重复文件。后续可以提供两种模式：

- 快速模式：采样 hash。
- 精确模式：完整 hash。

### 7.5 失效素材检查

难点：

用户可能在 AssetVault 外部移动或删除文件，数据库索引会变成悬挂状态。

方案：

提供失效检查功能，批量检测 `assets.path` 是否仍存在于磁盘，并把缺失素材标记出来，供用户集中处理。

### 7.6 AI 功能边界

难点：

很多 AI 项目容易变成聊天壳，但对业务价值不够直接。

方案：

AssetVault 的 AI 不做聊天，而是服务素材管理：

- 自动标签
- 自动描述
- 自然语言搜索
- 智能分类
- 相似素材推荐

当前实现先用本地规则引擎保证稳定可演示，接口设计保留 OpenAI Compatible API 配置，后续可以替换为真实 Embedding 和多模态模型。

## 8. API 设计亮点

后端 API 按领域拆分：

- `/auth`：登录和认证。
- `/folders`：素材目录管理和扫描。
- `/assets`：素材列表、详情、更新、收藏、评分、备注。
- `/tags`：标签管理。
- `/projects`：项目和项目素材引用。
- `/stats`：统计面板。
- `/settings`：系统设置、AI 配置、数据库备份。
- `/search`：自然语言素材搜索。
- `/ai`：智能分析。
- `/trash`：回收站。
- `/tasks`：任务状态。

讲解重点：

- API 按业务领域拆，不按页面拆。
- 前端页面可以复用同一组 API。
- 扫描和 Embedding 使用 PostgreSQL 持久化任务队列，通过行锁原子领取，并记录尝试次数、心跳和执行结果。

## 13. 高频面试追问

### Q1：为什么不用直接上传文件到服务器？

因为目标用户的素材通常体积大、目录结构复杂，而且已经存在本地硬盘中。直接上传会造成复制成本、存储压力和隐私问题。AssetVault 采用只建索引的方式，保留原始目录结构，系统只维护元数据和缩略图。

### Q2：如何保证删除安全？

删除默认只是软删除数据库索引，进入回收站，不删除真实磁盘文件。用户可以恢复索引，也可以永久删除索引。这个设计避免素材文件被误删。

### Q3：搜索为什么会比 Windows 搜索快？

因为 AssetVault 在扫描阶段已经把路径、文件名、类型、标签、备注等信息写入数据库。搜索时查数据库索引，不需要每次递归遍历磁盘。后续还可以通过数据库索引、全文检索和向量检索继续优化。

### Q4：AI 功能现在是真 AI 吗？

智能分析支持 OpenAI Compatible API，但结果先作为建议并明确来源，远端失败不会静默降级。自然语言搜索使用 BAAI/bge-m3 生成向量，存入 PostgreSQL pgvector，并与关键词排名进行 RRF 融合。

### Q5：如果素材库有几十万文件怎么办？

需要从三方面扩展：

- 调整 PostgreSQL 组合索引和 pgvector HNSW 参数。
- 扫描、缩略图和 Embedding 生成迁移到持久化任务队列。
- 搜索侧增加离线评测和分区策略，避免一次性加载过多候选。

### Q6：为什么不用 MongoDB？

素材、标签、项目引用、用户、收藏等关系比较明确，关系型数据库适合表达多对多关系、分页查询和统计聚合；pgvector 又让业务数据和 BGE-M3 向量在同一事务边界内管理。

### Q7：这个项目最大的工程难点是什么？

不是单个算法，而是把文件系统、数据库索引、前端交互和 AI 能力组合成稳定的产品闭环。尤其是“不移动原文件”的设计，需要处理失效索引、回收站、重复检测和备份等边界问题。
